# AGENT_ACTIVITY_UI_SPEC.md

## Project
Agent Activity UI enhancement for the Project Tracker application.

This specification defines the UI and UX requirements for exposing **agent ownership, agent workload, and live execution state** inside the Project Tracker board so the user can visually understand:

- which agent owns each task
- what each agent is currently doing
- which agents are active, blocked, idle, or waiting
- how Alfred is orchestrating work across the system

This spec is designed to work with the **existing project and task dialogs** shown in the current UI.

---

## Design Review Based on Current Dialogs
The current dialogs already establish a simple, clean design language:

- neutral modal surfaces
- understated borders
- navy primary action button
- standard form layout
- existing task fields for Priority, Type, Assignee, Column, and Sprint

Because of that, this enhancement should **extend the current design system rather than replace it**.

### Important implication
The current **New Task** dialog already includes an **Assignee** field. That means agent ownership can be introduced in a way that feels natural and low-friction:

- Alfred can automatically populate **Assignee** with the selected OpenClaw agent
- the UI can display agent ownership directly from that field
- the task dialog can be enhanced later to make agent selection easier, but no major conceptual change is required now

No major change to the earlier design direction is required. The new agent activity layer should remain visually consistent with the existing dialog styling.

---

# 1. Objective
Add a clear, modern, and lightweight **agent activity layer** to the Project Tracker UI so the board becomes a visible operations dashboard for OpenClaw.

The user should be able to answer these questions at a glance:

1. Which agent owns this task?
2. Which agent is actively working right now?
3. Which agents are idle or waiting?
4. How many tasks is each agent carrying?
5. What is Alfred coordinating across the board?

The design should feel like a **live mission control overlay** on top of the existing kanban board, without overwhelming the user.

---

# 2. UX Goals
The agent activity UI must:

- make agent ownership immediately visible on task cards
- make current system activity visible without opening each task
- preserve board readability
- avoid clutter and over-decoration
- remain compatible with the current Project and Task modal styling
- support future automation where Alfred creates and assigns tasks automatically

---

# 3. Core UI Features
The enhancement consists of four primary UI layers:

1. **Agent identity on each task card**
2. **Active Agents summary panel**
3. **Agent status indicators**
4. **Task-level activity metadata**

To avoid conflict with column status colors, agent colors must be subtle and secondary.

### Recommendation
Use **neutral task cards with workflow-state colors as the primary status cue**, and use agent accents only in badges, pills, dots, or mini-panels.

Suggested agent accent palette:

### Rule
Agent accent color must **never overpower the task status color**.

Example:
- task column status remains primary
- agent chip uses lighter tint and smaller footprint

# 9. Task-Level Activity Indicator
The UI should optionally show the current state of the assigned agent on that task.

## Supported Activity States
- `active`
- `waiting`
- `blocked`
- `reviewing`
- `idle`

## Visual Behavior
Use a tiny status dot or micro-badge alongside the agent chip.

Examples:
- green dot = active
- amber dot = waiting/reviewing
- red dot = blocked
- gray dot = idle

## Optional Text Examples
- `Working now`
- `Waiting on review`
- `Blocked`
- `Last update 5m ago`

### Rule
This metadata should remain compact. Do not turn task cards into logs.

# 17. Accessibility Requirements
Agent activity UI must be accessible.

## Requirements
- do not rely on color alone to indicate agent state
- include text labels such as `Active`, `Blocked`, `Idle`
- maintain readable contrast on chips and badges
- ensure screen-reader friendly labels for emoji/icon-only controls
- preserve keyboard navigation for task cards and filters

### Example accessible label
`Assigned to Pixel, frontend agent, status active`

# 20. Empty and Edge States
The UI must handle the following cases gracefully.

## Unassigned Task
If a task has no assigned agent:
- show `Unassigned`
- use neutral styling
- optionally show `Alfred pending assignment`

## Idle System
If no agents are active:
- Active Agents panel should say `No active agents right now`
- idle agents may still be listed in muted form

## Blocked Task
If a task is blocked:
- task activity line should show `Blocked`
- optional tooltip or details link may explain why

## Multi-Agent Work
If a task conceptually involves multiple agents, the system should prefer:
- splitting into subtasks
- assigning one primary agent per task

If multi-agent display is unavoidable:
- show primary owner first
- secondary agents may appear as small stacked chips or `+2`

# 22. Functional Constraints
This enhancement is focused on visibility and orchestration clarity.

It should not require a full redesign of the appвЂ™s data model to be useful.

Initial implementation should be possible using:
- existing assignee support
- additional agent metadata
- board-level summary data from Alfred

The feature should degrade gracefully if live activity data is incomplete.

---

# 23. Acceptance Criteria
The agent activity UI enhancement is complete when all of the following are true:

1. Each task card visibly shows the assigned OpenClaw agent.
2. The board includes a visible Active Agents summary area.
3. Agent rows/cards show current state such as Active, Waiting, Blocked, Reviewing, or Idle.
4. The task dialog supports selecting an OpenClaw agent in the Assignee field in a clearer, agent-aware way.
5. Agent activity is displayed without cluttering the board.
6. The design remains visually consistent with the current modal and board styling.
7. The UI remains responsive and accessible.
8. The board clearly communicates what Alfred is coordinating and what each agent is doing.
